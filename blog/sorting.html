House in Order

Bubble Sort, Merge Sort, and Quick Sort in JavaScript

January 6, 2016

The study of algorithms is an important topic in computer science, and an optimal algorithm can be much more important for the efficiency of a program than raw computing power. There are many types of algorithms used for a wide range of problems and there are many data structures that can be used to organize large amounts of data. In this article, I will only scratch the surface of an immense field by introducing three sorting algorithms which at their core deal with putting the elements of a list in a specific order. Bubble Sort, Merge Sort, and Quick Sort will provide a fundamental introduction to several algorithmic concepts. For the purposes of this article, I will treat the data as an array, which can be accessed via indices, as opposed to a sequentially-accessed linked list or other structure.

In this article, as I go through the three sorting algorithms, I will also discuss their common or uncommon properties including computational complexity, stability, memory usage, and adaptibility.

Bubble Sort is a simple sorting algorithm that works by comparing two items at a time in a list and swapping them if they are in the wrong order. It starts with the first element and compares it to the second. If the first is larger than the second, they are swapped. It then compares the second item to the third, the third to the fourth, etc, until the greatest value has sunk to the end of the list. Afterward, the process starts over again from the beginning and ends when the second-to-last element is compared, because the last element is already set in its final position. The process repeats until all of elements are in place.

We can look at a simple five item array to provide a visual for Bubble Sort.

array = [7, 1, 4, 9, 2]
// the first index is compared to the second
// since 7 > 1, they are swapped
[1, 7, 4, 9, 2]
// next, 7 > 4
[1, 4, 7, 9, 2]
// however, 7 < 9 so no swap occurs at this step
[1, 4, 7, 9, 2]
// 9 > 2, so 9 moves to the last index
[1, 4, 7, 2, 9]

At this point, the process repeats for the first four elements only, because the highest element, the 9, is already in its final position at the end of the list.

// 1 < 4, so no swap
[1, 4, 7, 2, 9]
// 4 < 7, so no swap
[1, 4, 7, 2, 9]
// 7 > 2, so they swap
[1, 4, 2, 7, 9]

And finally:

// 1 < 4, so no swap
[1, 4, 2, 7, 9]
// 4 > 2, swap
[1, 2, 4, 7, 9]
// 1 < 2, so no swap
[1, 2, 4, 7, 9]

Here is the basic code for Bubble Sort in JavaScript:

function bubble(array){
  var length = array.length;
  for (var i = 0; i < length; i++) {
    for (var j = 0; j < (length - i - 1); j++) {
      //if first element is greater than next element
      if (array[j] > array[j+1]) {
        //swap elements by creating temp var
        var temp = array[j];
        array[j] = array[j+1];
        array[j+1] = temp;
      }
    }
  }
}

The nested for loops above are a red flag when it comes to describing how expensive the algorithm is. The first time Bubble Sort makes its way through the array, it makes (n-1) comparisons between pairs, where N is the length of the array. The second time, it makes (n-2) comparisons since the final element is already in place. Calculating out the number of steps, we get
(n-1) + (n-2) + (n-3) + ... + 1 = n*(n-1)/2 = n2/2 - n/2 comparisons. For large datasets and independent of computing power, this algorithm as has a big O complexity of n2. Thus, as the length of the array grows, the proportional running time of this algorithm in the worst-case is quadratic.

Bubble Sort is a stable algorithm, which means that two elements that have the same value would remain in the same order relative to each other throughout - they would never cross paths or be swapped. Bubble Sort's memory usage is no larger than the original array, because the swaps are essentially taking place within the array. One redeeming feature of Bubble Sort is that it can be tweaked to detect if a list is already sorted or mostly sorted,(by adding a boolean variable to check if any swaps are made in one pass) in which case it will save the effort of continuing to go through the data if it is already in order. Not all algorithms are able to adapt in this manner! Unfortunately, due to it's slow average running time, and due to a large number of other sorting options available, Bubble Sort is rarely used in practice anymore. Next up, Merge Sort!

Merge Sort is an efficient sorting algorithm that runs faster than Bubble Sort's quadratic time in even the worst-case. It works by dividing up the data into halves until each element is in its own sublist. Then the algorithm recursively merges subjlists back together into sorted sublists until there is only one list left. Here are the steps that Merge Sort takes when it sorts a list:

array = [7, 1, 4, 5, 10, 2, 3, 9]
// divide the list into two sublists
7 1 4 5      10 2 3 9
// divide sublists into smaller sublists
7 1    4 5    10 2   3 9
// and divide those...
7  1  4  5  10  2  3  9
// now, "merge" the first two sublists into a sorted pair
1 7   4  5  10  2  3  9
// and merge the other pairs as well
1 7    4 5    2 10   3 9
// now, merge the sublist pairs into sorted quads
1 4 5 7      2 3 9 10
// finally, merge the two quads back into a sorted list
1 2 3 4 5 7 9 10

The merging works by simply comparing the first element in each sublist against one another. If the first element from the left is smaller, then the next comparison will be between the second element from the left sublist and the first from the right. The sort can be coded to be stable by always taking the element from the left sublist when the values are equal - that way the original order does not change.

The code for Merge Sort is as follows:

function merge(left, right) {
  var result = [];
  //do comparisons while both sublists still have elements
  while (left.length && right.length) {
    if (left[0] <= right[0]) {
      result.push(left.shift());
    } else {
      result.push(right.shift());
    }
  }
  //once one sublist is empty, only take elements from other list
  while (left.length)
    result.push(left.shift());
  while (right.length)
    result.push(right.shift());

  return result;
}

function mergeSort(array) {
  if (array.length < 2) {
    return array;
  }
  //divide list in half
  var middle = Math.floor(array.length/2);
  var left = array.slice(0, middle);
  var right = array.slice(middle);

  //mergeSort will continue to be called
  //dividing up the sublists
  //until the sublists are < 2 in length
  return merge(mergeSort(left), mergeSort(right));
}

The mergeSort function is responsible for recursively splittng up the original data in sublists until all sublists are less than length 2. The merge function compares the first index of each sublist against each other. Once one list is empty, it knows to just pull the remaining elements from the non-empty sublist.

The merging of the sublists takes linear time, so n where n is the length of the array. If T(n) is the number of comparisons to mergeSort an input of size n, that should be equal to T(n/2) + T(n/2) + n for the left half, right half, and merging, respectively. If we calculate out this recursive relation, we eventually end up with T(n) = n + nlogn = O(nlogn). Merge Sort works in linearithmic time in the worst case, faster than Bubble Sort's quadratic time.

One major drawback to Merge Sort is that it takes up a lot of memory; the algorithm as it was originally introduced creates new space for all of its elements as it is merging its sublists. The extra memory usage is n, the length of the original data. (As a reminder, Bubble Sort sorted the items in place.) As mentioned previously, Merge Sort is usually coded as a stable algorithm. As long as the element from the left sublist is taken in the event of a tie, no equal valued elements will cross paths during the sort. As it is coded above, there is limited adaptability. So if the dataset is already in order, the functions above will still go through the process of dividing up the original dataset and merging.

Merge Sort is a very important sorting algorithm today. Although it was created in 1945 originally, it is still widely used. Many tweaks have been implemented to improve upon Merge Sort's initial deficiencies, and many newer sorting algorithms either use Merge Sort or features of Merge Sort for certain datasets. For example, Timsort is a hybrid algorithm that combines ideas from Merge Sort as well as Insertion Sort, and it is used as the default built-in sort for Android, Java, and Python. Oddly enough, different languages have different default sorting algorithms under the hood, or may use multiple by default depending on the situation. Java does use Merge Sort for objects. C++ and Python use Merge Sort when a stable sort is needed. And Firefox uses Merge Sort for JavaScript. Merge Sort is a good choice for linked lists, for example, where the data can only be accessed sequentially, and some other sorting algorithms may be unable to sort linked lists altogether. Of course, a programmer can always code a sorting function however she wants to in order to avoid using the built-in sort for that language!

Lastly, I will talk about Quick Sort. Quick Sort is often faster than Merge Sort in practice when sorting arrays, and it uses less memory than Merge Sort. As you will see, it is not a stable sort, which means that equal-valued items can cross each other from their initial position. Chrome uses Quick Sort for JavaScript, and Unix, C, Python, Java, and Matlab all use Quick Sort in certain situations as well.

Quick Sort works by partitioning the original array around a pivot point. For the algorithm to work well, the pivot element ought to be close to the median of all the values in the array. (There are minor coding improvements to assure a good value without taking up too much time.) Once a pivot point is selected, the algorithm starts from both ends of the array and moves inward. When an element from the left counter is found that is greater than the pivot element, AND an element on the right is found to be less than the pivot element, they are swapped. When the left counter and right counter cross, the pivot is swapped with the right counter (now on the left). At this point, all items to the left of the original pivot will be less than the pivot, and all items to the right will be greater. Now, Quick Sort is performed on each half, recursively until the halves are less than two in length. Here is what it looks like:

[5, 2, 9, 0, 7, 3, 4, 6, 1]
// 5 is selected as the pivot element
// the left mark begins at index 1, moving to the right
// the right mark begins at the last index, moving left
[5, 2, 9, 0, 7, 3, 4, 6, 1]
// 2 < 5, so the left mark moves right one spot
[5, 2, 9, 0, 7, 3, 4, 6, 1]
// 9 > 5, so the left mark stays put
[5, 2, 9, 0, 7, 3, 4, 6, 1]
// 1 < 5, so the right mark and left mark switch places
[5, 2, 1, 0, 7, 3, 4, 6, 9]
// the process continues
// 0 < 5 and 6 > 5, so both marks tick
[5, 2, 1, 0, 7, 3, 4, 6, 9]
[5, 2, 1, 0, 7, 3, 4, 6, 9]
// 4 < 5 and 7 > 5, so they are swapped
[5, 2, 1, 0, 4, 3, 5, 6, 9]
// now the marks cross
[5, 2, 1, 0, 4, 3, 7, 6, 9]
// so the pivot is switched
[3, 2, 1, 0, 4, 5, 7, 6, 9]

One quick technique is to actually shuffle the array at the beginning of the algorithm, which (almost) guarantees that the array is not already sorted. If the array were already sorted, the pivot would consistently be the lowest element, and the algorithm would degrade to quadratic time.

It is important to realize that there are dozens of sorting algorithms, and many improvements have been made upon existing ones. However, Quick Sort and Merge Sort have each stood the test of time and are both very common today - with implementation fixes to try and ensure best possible run times.
